# Kano Model

## Overview

Kano model for classifying features as basic, performance, or delight—and applying it to Salesforce roadmaps.

## Purpose

Prioritize features by customer satisfaction impact, not only MoSCoW urgency.

## Why It Matters

Teams over-invest in delighters while basic expectations remain unmet.

## Business Context

Satisfaction is non-linear: missing basics causes dissatisfaction; delighters create loyalty when basics are met.

## Salesforce Context

Examples: basic = create/view Case; performance = faster search; delight = proactive AI summary (validate adoption).

## Core Concepts

| Category | Meaning | Salesforce example |
|----------|---------|-------------------|
| Basic | Expected; absence causes dissatisfaction | Login, create Account |
| Performance | More is better | Report speed, mobile offline |
| Delight | Unexpected positive surprise | Proactive case suggestion |
| Indifferent | Low impact | Rarely used admin cosmetic |
| Reverse | Some users dislike | Forced workflow change |

## Key Terminology

| Term | Definition |
|------|------------|
| Delighter | Feature exceeding expectations |

## Frameworks and Models

Use in roadmap workshops after MoSCoW for Could/Should ordering.

## Enterprise Best Practices

- Achieve basic parity before delighters
- Validate delighters with user research—not assumptions
- Consider adoption cost of delight features

## Common Mistakes

- AI features as delighter without data trust foundation

## Anti-Patterns

- Kano classification without user input

## Decision Guidelines

Enhancement programs: Kano for backlog ordering. Greenfield: MoSCoW first, Kano within releases.

## Real-World Examples

Dealer portal: basic = submit case; performance = status updates; delight = predicted resolution date (if validated).

## Industry Considerations

Healthcare: clinician time-saving is performance, not delight.

## AI Guidance

Do not classify features as delighters without stakeholder validation.
